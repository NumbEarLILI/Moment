# 备份恢复与「老手帐 + 新碎片」身份模型设计

## 1. 为什么会有「ID 冲突」？

这不是随机 Bug，而是**数据模型里混用了两种不同含义的 id**：

| 含义 | 实际是什么 | 本应满足的性质 |
|------|------------|----------------|
| **Room 表主键** `fragments.id` | 本机 SQLite 自增行号 | 仅在本机、当前库内唯一；可被回收、迁移、重装后变化 |
| **手帐里保存的** `sourceFragmentIds` | 备份/恢复时写进 `diary.json` 的「指向某条碎片的句柄」 | 恢复后仍要能**稳定**指向「那是旧稿里的第几条」，且与**后来新拍的碎片**绝不混用 |

NAS 当前只恢复 **手帐 JSON**，**不恢复** `fragments` 表。于是：

- 备份里写着：「第 1 条 plog = fragment **id = 1**」。
- 本机库里根本没有 id=1 这一行，或新用户第一条拍摄恰好插入 **id = 1**。
- 详情/合并逻辑用 `getFragmentsForSourceIds([1])` 取行 → 取到的是**新碎片行** → 旧手帐卡片上出现新图。

**结论：冲突来自把「本地自增主键」当成了「跨恢复、跨会话的稳定业务 id」。**  
占位行、过滤当日碎片等做法都是在**补救这一语义错误**；要治本，必须重新定义「手帐引用的碎片身份」或「备份里必须包含什么」。

## 2. 设计目标（老手帐 + 新碎片）

1. **恢复后**，未经过用户「合并进这篇手帐」的新碎片，**绝不能**通过任何路径（详情展示、选点刷新、规则生成写回等）写入**该篇已存手帐**的持久化字段。
2. **备份恢复**后，旧稿时间线在**无本地 `fragments` 行**时仍能展示（文案 + 分卡图已由手帐自身字段承载）。
3. **用户明确执行「合并新碎片」**时，才把手帐时间线与新拍摄的**业务身份**建立关联，且关联应**可序列化到备份**，重装后不依赖「碰巧主键相等」。
4. 尽量少改 UI 心智：「碎片池」仍是按日拍摄；「手帐」仍是已保存的一篇文档；界线在**数据与用例**，而不是再多一层神奇启发式。

## 3. 三种架构路线（对比）

### 方案 A — 备份包包含「碎片实体」

- **做法**：NAS（或未来本地备份）在每条手帐目录下增加 `fragments/*.json`（或一个 `fragments.json` 列表），恢复时先插入/替换 `fragments` 行，**主键与备份一致**（或显式 `WITHOUT ROWID` + 迁移策略），再写入手帐。
- **优点**：`sourceFragmentIds` 继续可用；与现有代码路径最接近；语义直接——备份是「手帐 + 其引用的素材」的快照。
- **缺点**：包体积变大；需定义碎片与手帐的版本、冲突（同机已有碎片 id）；删除需约定是否「只删手帐不删素材」。

### 方案 B — 手帐只引用「稳定业务 id」（推荐长期方向）

- **做法**：为每条「可进手帐时间线的条」引入 **`fragmentStableId: UUID`（或 ULID）**，与 Room 自增 `id` 分离。  
  - `fragments` 表：`localId`（自增）+ `stableId`（唯一）。  
  - `DiaryEntry.sourceFragmentIds` 存 **`stableId` 列表**（序列化为 string）。  
  - 备份只序列化 `stableId` + 手帐内嵌的 story / 图映射；恢复时可无本地行，仅靠手帐内嵌展示；合并新碎片时把新拍摄的 `stableId` **追加**进列表。
- **优点**：从根本上消灭「自增 id 当外键」；跨设备、跨恢复语义清晰。
- **缺点**：需一轮数据迁移与全链路改造（DAO、合并、测试）。

### 方案 C — 手帐内完全自包含时间线（弱化对 `fragments` 的依赖）

- **做法**：已保存手帐的每一条 plog 以 **`PlogSegment` 嵌套结构**存进 DB（`segmentId` + 文案 + 图 URI + 可选 `linkedLocalFragmentId`）。备份只带嵌套结构；详情优先用嵌套数据渲染，`fragments` 仅在有「仍可编辑的关联拍摄」时作为补充。
- **优点**：恢复后零依赖本地碎片表也能 100% 还原展示。
- **缺点**：与当前「手帐字段 + 外链碎片」的双存储模型重复；合并逻辑要写清楚何时同步回嵌套结构。

## 4. 推荐路线（分阶段，可落地）

**短期（保持当前产品形态，降低再次改错概率）**

1. **语义写清**：文档与代码注释中明确：`fragments.id` 仅为本地主键；**禁止**把它当作备份/恢复中的唯一业务标识。
2. **备份契约**：在 NAS 文档与 manifest 中写死——当前 v1 **仅手帐**则必须在恢复时执行「占位/预留 id」或 **方案 A 的最小子集**（至少把备份里出现的 fragment id 写成空行占位），直到升级为完整方案。
3. **已存手帐的自动刷新**（如选点）：只允许更新**已纳入该手帐锚点集合**内的数据；**禁止**调用会把「当日全部碎片」并入草稿的生成路径（已通过 `appendDayFragmentsOutsidePrior` 等约束落实，需在 spec 层写为不变量）。

**中期（建议优先实施 方案 A）**

- 备份为「一篇手帐 = diary.json + 该手帐引用过的碎片快照（可选仅锚点 id 对应的最小字段）」。
- 恢复顺序：`fragments`（带 id）→ `diaries`，保证 `sourceFragmentIds` 与真实行一致。
- 新碎片仍为自增；只要备份里的 id 块已被恢复行占据，就不会与新插入抢号。

**长期（方案 B）**

- 新数据一律 `stableId`；迁移脚本为旧库生成 UUID 并写回；`sourceFragmentIds` 逐步改为 string id 列表或并行双字段过渡期。

## 5. 「合并新碎片」在模型上应该怎样定义？

**事件**：用户从某篇已保存手帐进入「合并新碎片」并成功保存。

**效果（不变量）**：

1. 手帐的「锚点集合」**显式扩大**：包含新碎片对应的 **稳定身份**（方案 B）或 **已存在且与备份一致的本地 id**（方案 A）。
2. 此前未合并的同日碎片，**不得**因其它副作用（地点刷新、规则重算）自动进入 `sourceFragmentIds`。
3. 生成草稿用例必须区分：  
   - **打开预览 / 用户意图合并**：允许 `appendDayFragmentsOutsidePrior = true`（在 prior 已正确传入的前提下）。  
   - **被动刷新已存手帐**：`appendDayFragmentsOutsidePrior = false`。

## 6. 验收标准（建议作为后续实现的测试清单）

1. 仅恢复手帐、无碎片表行：详情页时间线与恢复前备份一致；**新增拍摄**后，详情**不出现**新图，直至用户走合并并保存。
2. 恢复后带碎片快照（方案 A）：`sourceFragmentIds` 与本地行一一对应，新碎片 id 与备份 id **无重叠**。
3. 若实施方案 B：同一手帐在 A 设备保存后在 B 设备恢复，`stableId` 列表一致即时间线一致，与 B 设备本地 `fragments.id` 数值无关。

## 7. 与当前代码的关系

当前仓库中的占位碎片、合并限制、导航 diaryId 等，属于在**未升级备份格式与身份模型前**的防御性实现；**本 spec 定义替代它们的正统方向**。待选定方案 A 或 B 并完成实现后，应逐步删除仅用于弥补「只恢复手帐、无碎片」的临时手段，并补充端到端测试（恢复 → 新拍 → 详情 / 合并）。
